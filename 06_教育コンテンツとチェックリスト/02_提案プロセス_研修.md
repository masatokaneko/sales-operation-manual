# 提案書作成研修

## 📋 研修の目的

この研修は、顧客に響く効果的な提案書を作成するためのスキルを身につけることを目的としています。
提案書は単なる商品説明ではなく、**顧客のビジネス課題解決と目標達成を約束する契約書の下書き**です。

---

## 🎯 1. 提案書作成の基本原則

### 1.1 必ず守るべき4つの原則

1. **必ず「提案書」を作りましょう**
   - 口頭説明だけでは伝わらない複雑な内容を整理
   - 顧客内での共有・検討のための資料として活用

2. **読み手と利用目的を明確にしましょう**
   - 誰が読むのか（技術者、経営層、決裁者等）
   - 何のために使うのか（プレゼン、社内回覧、稟議書等）

3. **ストーリーを明確にしましょう**
   - 論理的な流れで課題から解決策まで一貫性を保つ
   - 読み手が納得できる筋道を構築

4. **「買いたい」と思わせる内容にしましょう**
   - 顧客のメリットを具体的に示す
   - 競合他社ではなく「なぜScalarなのか」を明確化

## 🔄 2. 提案書の目的と顧客視点

### 2.1 目的の再定義

| 従来の考え方 | あるべき考え方 |
|-------------|---------------|
| 商品のスペック紹介 | **顧客のビジネス課題解決と目標達成を約束する契約書**の下書き |
| 何を売るか | **顧客にどんな価値をもたらすか**に焦点 |

### 2.2 顧客視点の徹底

- **KPI・業界トレンド・競合動向**を常に念頭に置く
- **キーパーソンの個人的な目標**を理解する
- **WIIFM (What's In It For Me?)**：顧客にとっての利益を明確にする

### 2.3 差別化の明確化

- **技術力とビジネス知見の融合**による唯一無二の価値を示す
- **Because文**：なぜこの提案が成功するのかの根拠をロジカルに提示

---

## 📋 3. 提案書の構成

### 3.1 基本構成（4部構成）

| 章 | 目的と内容 | 構造的なキーポイント |
|---|------------|---------------------|
| **Executive Summary** | 最重要な「結論」を最初に提示<br>課題・解決策の概要・期待されるビジネスインパクトを1〜2枚でまとめ | **「30秒で理解できるか」**<br>多忙な経営層はここしか見ない可能性 |
| **I. 顧客の現状と課題 (As-Is)** | 客観的なデータや市場動向に基づく課題認識<br>顧客が気づいていない潜在的課題にも切り込み | **「痛みの共有」**<br>課題認識が深ければ深いほど顧客は引き込まれる |
| **II. 提案のビジョンと解決策 (To-Be)** | 課題解決後の理想的な姿を明確に描く<br>実現に向けた具体的ソリューションの全体像を提示 | **「ビジョンの提示」**<br>未来の成功像を顧客と共有 |
| **III. ビジネスインパクトとROI** | 導入後の定量的・定性的効果を具体的に試算<br>提案の費用対効果を証明 | **「財務的な説得力」**<br>投資に見合うリターンをKGI/KPIで明確に示す |
| **IV. 実行計画とリスク** | 導入ロードマップ・体制・マイルストーンを具体的に提示<br>リスクとその回避策も提示し信頼性を高める | **「実現可能性」**<br>絵に描いた餅ではなく現実的な実行力を示す |

## 🚀 4. Scalarならではの提案書作成ポイント

### 4.1 必須要素

- **ビジネスインパクトは必要**
- **課題→ソリューション→Scalar製品が必須**という流れを構築
- 製品を使うことが必須になっていないと、後から外されるリスクあり

### 4.2 Scalar提案書の例示

**課題設定**
```
顧客は外注依存になっており
「コスト高・改修に時間がかかる・ビジネスチームから不評」
```

**ソリューション**
```
AI駆動開発 × ScalarDBで
「開発の高速化」+「改修に適した基盤整備」を実現

フロント開発：AI駆動
バックエンド・DB開発：ScalarDB with AI駆動が最適
```

**注意点**
```
フロントだけの改修は場当たり的対応にしかすぎず、
バックエンド+インフラも一緒にやらないと効果が限定的
→ インフラの刷新を並行して実行
```

**実行プロセス**
```
移行システム開発 → 移行システム本番化 → 本番運用

基本的には「学習 → 実践 → 蓄積」の順番
内製化チームが主導でものを作っていく
```

**費用対効果**
```
移行システム開発段階で数値を取得
開発期間を半分に削減することを目標

費用：ライセンス7pods × 228万円 = 1,596万円
```

## 🛠️ 5. スライド作成の正しい手順

### 5.1 ✅ 正しい手順（6ステップ）

> [!IMPORTANT]
> **スライド作成するのは最後！**

1. **（思考）** 誰に・どのようになってもらいたくて資料を作成するかを決める
2. **（思考）** それに従って、どのようなストーリーであればそれが伝わるかを考える
3. **（Docs）** シンプルな文章で、ブレッドポイントでメッセージを作成する
4. **（Docs）** 各メッセージはどのようなコンテンツがあれば伝わるかをブレッドポイントの下に列挙する
5. **（紙）** 各メッセージをどのように表現すればいいかを手書きで作成する
6. **（Slide）** スライドを作成する

### 5.2 ❌ やってはいけないこと

- **A.** 既存のスライドの切り貼りでなんとなく資料を作る
- **B.** どんな資料にするかを明確にしないでとりあえずSlideを作り始める
- **C.** メッセージがよくわからない資料を作る（1 slide 1 message違反）

## 📚 8. 構造化思考の基礎

### 8.1 構造化のメリット

1. **問題解決の漏れやダブりを回避**しやすくなる
2. **問題の原因に対して適切な対処**ができるようになる
3. **問題解決の優先順位**がつけやすくなる
4. **情報共有やコミュニケーション**がしやすくなる

### 8.2 構造化の5ステップ

1. **全体を定義する**
2. **構成要素に分解する「切り口」を探す**
3. **切り口にそって構成要素に分解する**
4. **構成要素一つ一つの特徴を分析する**
5. **全体と部分の間で働くメカニズムを発見する**

### 8.3 重要な概念

#### MECE（ミッシー）
- **Mutually Exclusive, Collectively Exhaustive**
- 重なりなく、しかも全体として漏れのない部分の集まりでとらえること

#### So What? / Why so?
- **So What?**: 手持ちのネタから課題に照らした時に言えることのエキスを抽出
- **Why so?**: So What?した要素の妥当性を検証する作業

### 8.4 構造化の5つの方法

1. **要素分解による構造化**：要素の足し算で分解する方法
2. **因数分解による構造化**：要素の掛け算で分解する方法
3. **フレームワークによる構造化**：既存のフレームワークを利用する方法
4. **概念分解による構造化**：曖昧な概念をその構成要素によって分解する方法
5. **プロセスによる構造化**：物事のプロセスによって分解する方法

## 🎨 9. スライド作成のテクニック

### 9.1 基本ルール

- **「メッセージ」と「タイトル」は別物**
  - メッセージ：そのスライドで伝えたい主張
  - タイトル：グラフ・チャート・表の説明

### 9.2 メッセージの3条件

1. **1スライド1メッセージ**となっているか？
2. **明快な主張**がある（ポジションが明確）
3. **適度に短い**

### 9.3 作成時の留意点

- **グラフやチャートは最初に書くと「死にスライド」になる**
  - まずメッセージを決めてから、それを支える図表を作成
- **出所は先に書く**（忘れてしまうので）
- **言いたいことを書き出して、何が本当に言いたいことかを考える**
- **その一文で何を話しているのか伝わるようにする**
