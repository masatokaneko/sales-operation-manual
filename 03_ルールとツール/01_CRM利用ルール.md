# 📘 Salesforce登録ルール（営業活動全般）

## 1.3.1 登録の基本原則

### ✅ 必須登録ルール
- **全訪問・打合せはSalesforceに記録すること**
  - 訪問後24時間以内を目安に登録
  - 記録は「取引先」「商談」「取引先責任者」に紐付ける
- **議事録・資料はQuipで作成し、Salesforceに連携する**
  - 商談オブジェクトまたは取引先責任者オブジェクトにリンクを貼る
  - 関係者（社内外）をQuip上で@メンションすること

---

## 1.3.2 登録必須項目（未登録はStage進行不可）

### 基本情報

| カテゴリ  | 項目                     | 備考             |
| ----- | ---------------------- | -------------- |
| **商談情報**  | 商談名（命名規則：年度_顧客名_案件内容） | 例：FY25_トヨタ自動車_データ基盤刷新PoC |
| **金額情報**  | 数量×単価（直接金額入力禁止）        | Scalar製品単価表を参照 |
| **顧客情報**  | 取引先責任者（役職／部署／メール／電話番号）      | ファーストコール参加者は必須             |
| **商流情報**  | 商流区分（Direct or 代理店名）           | 代理店名を明記        |
| **提案状況**  | ステージ／次のステップ／Close予定（発注書受領予定日）    | 毎週更新必須         |
| **リンク情報** | Quip議事録・Box見積書リンク      | 空欄禁止           |
| **製品情報** | 製品（DB, DL, 3rdP, Service） | 対象製品を明確化 |
| **Type情報** | CNSL, POC, DEV, PROD, RNEW, ADD | 案件種別を明記 |
| **キャンペーン** | 主キャンペーンソース | マーケ起点案件は必須 |

### 取引先（Account）
- 正式名称、所在地、業種、企業規模を入力
- 顧客担当者が判明している場合、取引先責任者と紐付ける

### 取引先責任者（Contact）
- 氏名、部署、役職、メールアドレス、電話番号
- ファーストコールの参加者は必ず登録

## 1.3.3 商談名（案件名）命名ルール

### 基本フォーマット
> **「年度_顧客名_案件内容」**

### 🔹命名例
- `FY25_トヨタ自動車_データ基盤刷新PoC`
- `FY25_スタンレー電気_MBSE支援_開発環境導入`
- `FY25_三井住友海上_契約データ改ざん防止_提案`

### 各構成要素のルール

| 区分         | 内容                                  | 記載例                             |
| ---------- | ----------------------------------- | ------------------------------- |
| **① 年度**   | Salesforce上の会計年度（Fiscal Year）を基準 | `FY25`                          |
| **② 顧客名**  | 正式法人名を原則使用（略称可。ただし社内統一必須）           | `トヨタ自動車`, `Hino`, `SMBC`        |
| **③ 案件内容** | 商談の目的・スコープを端的に（15〜20文字以内）           | `PoC`, `本番導入`, `契約更新`, `AI開発支援` |

### 補足ルール

1. **複数の製品を含む案件**は、主要製品名を含める
   例：`FY25_ENEOS_ScalarDB本番導入`

2. **継続案件**の場合は末尾に `_Renewal` を付与
   例：`FY24_Stanley_MBSE支援_Renewal`

3. **複数部署を跨ぐ案件**は、括弧書きで部門名を記載
   例：`FY25_HINO(IT企画部)_データ統合PoC`

4. **パートナー経由の案件**は、末尾に `_via_パートナー名` を付与
   例：`FY25_SumitomoLife_ScalarDB_via_NTTDATA`

5. **同一顧客で複数商談が並行する場合**は、末尾に枝番をつける
   例：`FY25_Stanley_MBSE支援_01`, `FY25_Stanley_MBSE支援_02`

### ❌ 禁止例

| 誤った例                 | 問題点                  |
| -------------------- | -------------------- |
| `トヨタシステム案件`          | 年度が抜けている             |
| `FY25_TOYOTA`        | 内容が不明確               |
| `FY25_トヨタ_検討中`       | 不明瞭なステージ用語（「検討中」は不可） |
| `ToyotaPoC2`         | 英語・日本語・年度の混在／区切りなし   |

## 1.3.4 活動記録（Activity Log）

### 議事録
- 訪問日時、訪問先、参加者（顧客・自社両方）を明記
- 内容、ToDo、Next Actionを記載
- 顧客に送付した議事録と同一内容を登録
- Salesforce上では商談・取引先責任者の両方に紐付ける

### メール送付履歴
- 顧客に送付したメールの要約（3行サマリ＋ToDo）を活動メモとして残す

### 添付資料
- 提案資料、事例資料、議事録PDFなどをQuipまたはBoxリンクで登録
- ファイル名ルール：「YYYYMMDD_顧客名_議事録／提案資料」

---

## 1.3.5 商談ステージ別の登録タイミング

| ステージ                                    | Salesforceで登録・更新すべき情報                |
| --------------------------------------- | ------------------------------------ |
| **Stage 1: Assessment & Qualification** | 商談新規登録（顧客課題の認識・タイムライン）               |
| **Stage 2: Discovery**                  | ディスカバリーマップ、システムマップ、インフルエンスマップをQuip添付 |
| **Stage 3: Solution Development**       | PoC合意、提案スコープ、関連パートナー情報               |
| **Stage 4: Solution Presentation**      | 提案資料、見積書、競合比較内容                      |
| **Stage 5: Resolution**                 | 稟議・契約スケジュール、契約ドラフト                   |
| **Stage 6: Closed Won / Lost**          | 契約締結日、金額確定、失注理由                      |

> [!NOTE]
> 各ステージの詳細な定義と主要成果物については、[02_営業プロセス/00_商談活動プロセス全体像.md](../02_営業プロセス/00_商談活動プロセス全体像.md)を参照してください。

### ステージ変更時の必須事項
- **Sales Operationチームへの通知**：ステージ変更時は必ずSales Operationチームに通知
- **進捗根拠の提示**：商談ステージが進捗した根拠を明確に提示
- **変更履歴の記録**：Salesforceの活動履歴にステージ変更の根拠を詳細に記載
- **Deal Desk承認**：Deal Desk承認は商談ステージ進行には不要（Deal Deskは案件審査のみ）

---

## 1.3.6 登録責任とチェック体制

### 登録責任者
- **アカウントエグゼクティブ（AE）**：全体責任
- **SA（ソリューションアーキテクト）**：技術資料・提案内容を補足登録

### チェック責任者
- **セールスオペレーション**：登録漏れ、ステージの整合性、金額の整合性を毎週確認

---

## 1.3.7 更新ルールと報告サイクル

### 更新頻度
- **週次でステータス更新**
- ステージ移行時にはWPS（Win Planning Session）の議事録を登録

### レビュー対象
- 取引先責任者、商談、活動ログ、ToDo

---

## 1.3.8 重要事項（CFO指示）

> [!IMPORTANT]
> Salesforce情報は**投資家・取締役会報告の基礎データ**であるため、常に最新化が必要です。
> 
> **新規案件登録時の必須作業**
> - Boxフォルダ作成
> - 種別・完了予定日の登録
> - 主キャンペーンソースの設定
> - 取引先責任者登録（Gmail連携含む）
> 
> **情報更新の徹底**
> - 提案・見積・契約・請求の進捗をSalesforceで逐次更新
> - Box連携フォルダに提出書類を格納
> - 常に最新化を維持

---

## 1.3.9 運用ポイントまとめ

- Salesforce、Quip、Box、Freee、Slack Botなどの**全連携システムで共通命名を適用**
- 命名は「**誰が見ても識別できる・誤解を生まない**」ことを最優先
- AEが命名し、SA・セールスオペレーションが確認
- 命名規則を逸脱する場合は、必ずセールスオペレーションの承認を得る
