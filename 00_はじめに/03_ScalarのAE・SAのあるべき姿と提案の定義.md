# AE（Account Executive）とSA（Solution Architect）のあるべき姿

このドキュメントは、Scalarのエンタープライズセールスにおける **AE（Account Executive）** と **SA（Solution Architect）** の「あるべき姿（理想像）」を、実務的・文化的・評価的観点から体系化したものです。

---

# AE（Account Executive）のあるべき姿

## 1. 役割の定義

- 顧客の「経営課題」から出発し、Scalar製品を**本番システムで導入**いただくことを通じて、経営的インパクトを生み出すことを使命とする。
- 単なる営業担当ではなく、顧客の中期経営計画・組織構造・DXロードマップを理解した**事業変革パートナー**として機能する。
- 案件を“作る”営業ではなく、“戦略的に勝ちに行く”営業を目指す。

## 2. 主要ミッション

- 顧客アカウントの経営構造・組織体制・投資テーマを深く理解し、アカウントプランを主導して作成する。
- Stage2までは「課題探索（Discovery）」に集中し、Stage3以降は「価値提案（Solution）」に転じる。
- Deal Desk・WPS・Forecastなどのプロセスを自ら主導し、プロジェクトを経営層に責任を持って報告する。
- 営業活動を定量的に管理し、日次・週次・四半期レビューに耐えうる透明性の高いデータを提供する。

## 3. 行動原則

- 顧客の立場に立ちながら、常に「Why Now?」「So What?」「Next Step?」を意識する。
- **提案は「本番導入」を前提とした設計書**であり、曖昧な仮説ではなく、構造化されたストーリーを提示する。PoCの提案は行わない。
- 社内の各SAチーム・SalesOps・Corporateとの連携を怠らず、全体最適で案件を推進する。
- 予算・権限・課題・時期（BANT）の明確化を自らリードし、商談の進行を意図的に設計する。

## 4. 期待される成果

- 四半期ごとのアカウントプランを完成度高く更新し、重点顧客での受注確率を高める。
- Stage別コンバージョン率、Forecast精度、Quip登録率、議事録提出率などで高水準を維持。
- 提案書・見積・契約までの一貫したストーリーを構築し、顧客から「信頼される経営パートナー」として評価される。

## 5. 評価基準

- 受注金額・成約率などの定量指標に加え、以下を重点評価：
  - アカウントプランの完成度（定性・定量両面）
  - 提案ストーリーの質（Why/Howの明確性）
  - 社内連携（各SAチーム・Ops・Mgrとの協働度）
  - 顧客満足度・再提案率

# SA（Solution Architect）のあるべき姿

## 1. 役割の定義

ScalarのSAは、顧客の成功と製品価値の最大化を実現するため、以下の3つの専門チームに分かれて活動する。それぞれが技術ブランドの担い手であり、AEと協働して顧客への価値提供をリードする。

### 1.1 Discovery SA （課題発掘・技術検証担当）
- AEと共に商談の初期段階に参画し、顧客の技術的な現状（As-Is）と課題を深く理解する「技術的信頼の源泉」。
- 顧客のエンジニアと対話し、技術的な実現可能性を評価する。

### 1.2 Demo SA （プロダクトレッドグロース・デモ担当）
- 製品の価値を視覚的かつ体験的に伝える「デモとアプリケーションの構築者」。
- AI駆動開発などを活用し、顧客が実利用をイメージできる高レベルなデモ環境を構築・提供する。

### 1.3 Post-Sales SA （導入・実装支援担当）
- 受注後、顧客の実装プロジェクトに伴走し、技術的な成功を確実にする「実装パートナー」。
- スムーズな本番稼働と、その後の安定運用を支援する。

## 2. 主要ミッション

- **Discovery SA:** 顧客課題・システム構成・データ構造を理解し、最適なアーキテクチャの方向性を提案する。
- **Demo SA:** ほぼ実利用できるレベルのアプリケーションとして、製品がどのように動くかを示すデモを設計・構築・実行する。
- **Post-Sales SA:** Proof of Concept（PoC）やトライアルの設計・実行・検証を主導し、顧客の本番実装を成功に導く。

## 3. 行動原則

- **Discovery SA:** AEが構築する商談ストーリーを技術的に補完し、信頼性を裏付ける。
- **Demo SA:** 顧客に「なぜScalarでなければならないか」を視覚的に理解させるストーリーテリングを行う。
- **Post-Sales SA:** 製品の制約やリスクを正直に伝えつつ、実現性を担保したソリューション設計を行う。
- **全SA共通:** 知識を社内に還元し、他案件の再利用・横展開を促す。

## 4. 期待される成果

- **Discovery SA:** 質の高い技術ヒアリングによる、提案の方向性の明確化。
- **Demo SA:** 説得力のあるデモによる、顧客の導入意欲の向上と商談期間の短縮。
- **Post-Sales SA:** PoC成功率・実装成功率・本番安定稼働率の高水準での維持。

## 5. 評価基準

- **Discovery SA:** 技術提案の品質、AE/顧客からの信頼度。
- **Demo SA:** デモの品質と完成度、それが商談に与えたインパクト。
- **Post-Sales SA:** PoC/導入案件の成功率、顧客満足度。
- **全SA共通:** ナレッジ共有・社内教育への貢献度。

# AEとSAの協働モデル（理想形）

## 1. 商談初期（Stage1〜2）

- **AE:** 顧客課題の把握とBANT確認に集中。
- **Discovery SA:** AEに同行し、顧客の技術課題・現状を把握。技術的な観点から商談の確度を判断する。
- **Demo SA / Post-Sales SA:** 参加しない。

## 2. ソリューション設計（Stage3）

- **AE:** 提案骨子とビジネス価値を設計。
- **Discovery SA:** 技術構成、データ設計、アーキテクチャの方向性を提案。
- **Demo SA:** 必要に応じて、提案内容を補強するデモの準備を開始する。

## 3. 提案・交渉（Stage4〜5）

- **AE:** 経営・業務層へのプレゼン、価格交渉を担当。
- **Demo SA:** 技術層・情報システム部門に対し、具体的な製品デモを実施し、技術的優位性と実現性を示す。
- Deal Desk・WPSでAEと関係SAがセットで参加し、推進と牽制のバランスを取る。

## 4. 成約後（Stage6以降）

- **Post-Sales SA:** 導入・技術引継ぎ・サポートチーム連携を主導。本番導入のマイルストンとしてPoCが設定されている場合は、その実行を管理する。
- **AE:** 顧客満足度と継続・拡張提案を管理。
- **全員:** 「成功事例（Success Story）」をQuip／Boxに蓄積。

# 補足：理想的な姿勢の共通項（AE・SA共通）

- Scalarの価値は「製品単体」ではなく、「顧客の意思決定構造に変化をもたらすこと」で発揮される。
- AEは“顧客の経営構造”を理解し、SAは“顧客のデータ構造”を理解する。
- 両者が一体となって「Why Scalar」「Why Now」「Why This Way」を語れることが理想である。
