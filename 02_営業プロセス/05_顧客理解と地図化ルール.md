# 顧客理解と地図化ルール

## 1. 概要

AEは顧客の真の課題を把握し、関係者構造を理解するため、3つの地図（Map）を作成・維持する必要があります。これらの地図は、Stage1-2（Assessment/Discovery）で作成し、商談を通じて継続的に更新します。

---

## 2. ディスカバリーマップ（Discovery Map）

### 2.1 目的
顧客の課題構造を可視化し、表層課題と真因を分けて整理することで、Scalarが解決すべき本質的な課題を特定する。

### 2.2 作成タイミング
- **初回作成**：Stage1（Assessment）
- **更新**：顧客訪問毎に新情報を反映

### 2.3 構成要素

#### 表層課題（Surface Problems）
顧客が認識している直接的な問題
- システムの応答速度が遅い
- データ整合性エラーが頻発
- スケーラビリティの限界
- 運用コストが高い

#### 中間課題（Intermediate Issues）
表層課題の背景にある構造的問題
- アーキテクチャの設計上の制約
- レガシーシステムとの連携問題
- 技術的負債の蓄積
- 組織間の情報連携不足

#### 真因（Root Causes）
問題の根本原因
- データベースのACID特性の欠如
- 分散システムの一貫性保証の不在
- トランザクション管理の複雑性
- ビジネス要件と技術実装のギャップ

### 2.4 作成方法

1. **5 Whys分析**
   - 表層課題から「なぜ？」を5回繰り返し、真因を探る
   - 例：「決済処理でエラーが発生」→なぜ？→「在庫と決済の不整合」→なぜ？→...

2. **ペインチェーン分析**
   - 問題が引き起こす連鎖的な影響を可視化
   - ビジネスインパクトまでつなげる

3. **優先順位付け**
   - Impact（影響度）× Urgency（緊急度）マトリクス
   - Scalarで解決可能な課題を特定

### 2.5 Salesforce登録項目
- **Pain Points**フィールドに要約を記載
- **詳細はQuip**にDiscovery Mapドキュメントを作成しリンク

---

## 3. システムマップ（System Map）

### 3.1 目的
顧客の既存システム構成を把握し、どのシステムが課題の原因か、誰が関与しているかを理解する。

### 3.2 作成タイミング
- **初回作成**：Stage2（Discovery）でDiscovery SAと共同作成
- **更新**：技術検証やPoCを通じて詳細化

### 3.3 構成要素

#### システム構成図
- **アプリケーション層**
  - フロントエンド（Web/Mobile）
  - API Gateway
  - マイクロサービス

- **データ層**
  - 既存データベース（種類、バージョン）
  - データウェアハウス
  - キャッシュ層

- **インフラ層**
  - オンプレミス/クラウド環境
  - ネットワーク構成
  - セキュリティ要件

#### 課題マッピング
- 各システムコンポーネントに課題を紐付け
- ボトルネックの特定
- ScalarDB/DLの導入ポイント明確化

#### 関与者マッピング
- 各システムの管理責任者
- 開発チーム
- 運用チーム
- ベンダー関係

### 3.4 作成方法

1. **現状調査**
   - 既存システムのドキュメント収集
   - 技術担当者へのヒアリング
   - システム構成の図解

2. **課題の紐付け**
   - Discovery Mapの課題を各システムに配置
   - データフローに沿った問題分析

3. **ScalarDB/DL適用箇所の特定**
   - 置換可能なコンポーネント
   - 新規追加が必要な機能
   - 移行の優先順位

### 3.5 Salesforce登録項目
- **Technical Requirements**に要約記載
- **詳細はConfluence**にSystem Architecture文書として保存

## 4. インフルエンスマップ（Influence Map）

### 4.1 目的
意思決定者、利用者、反対勢力を明確にし、関係性を描くことで、効果的なステークホルダー戦略を立案する。

### 4.2 作成タイミング
- **初回作成**：Stage1-2で基本構造を把握
- **更新**：商談の進展に応じて関係性を詳細化

### 4.3 ステークホルダーの分類

#### 役割による分類

##### Champion（推進者）
- **定義**：プロジェクトを強力に推進し、社内調整を主導する人物
- **特徴**：
  - Scalarの価値を深く理解
  - 社内に影響力を持つ
  - 積極的に情報提供してくれる
- **アプローチ**：
  - 定期的な情報共有
  - 社内説得用の資料提供
  - 成功事例の共有

##### Coach（内部支援者）
- **定義**：裏付け情報を提供し、社内事情を教えてくれる協力者
- **特徴**：
  - 中立的だが協力的
  - 社内プロセスに詳しい
  - リスク情報も共有してくれる
- **アプローチ**：
  - 早期に見つけて関係構築
  - 定期的な1on1ミーティング
  - 情報の機密性を保証

##### User（利用者）
- **定義**：実際にシステムを使用する現場担当者
- **特徴**：
  - 現場の課題を最も理解
  - 技術的な詳細に関心
  - 変更への抵抗感を持つ可能性
- **アプローチ**：
  - 現場の声を丁寧に聞く
  - 具体的な改善効果を示す
  - 移行時のサポート体制を説明

#### 影響力による分類

##### Decision Maker（最終決裁者）
- CxOレベル
- 予算承認権限を持つ
- ビジネス価値を重視

##### Influencer（影響者）
- 決裁者に助言する立場
- 技術評価を行う
- 部門責任者

##### Blocker（反対勢力）
- 現状維持を望む
- 競合製品を推奨
- リスクを過大に評価

### 4.4 関係性の可視化

#### Power-Interest Grid
```
高 ┌─────────────┬─────────────┐
影 │   満足維持   │  緊密管理    │
響 │  (Monitor)  │ (Manage)    │
力 ├─────────────┼─────────────┤
   │  最小努力    │  情報提供    │
低 │  (Minimal)  │  (Inform)   │
   └─────────────┴─────────────┘
     低          関心度         高
```

#### 関係性マトリクス
- 各ステークホルダー間の関係（協力/対立/中立）
- 情報フローの経路
- 意思決定の流れ

### 4.5 戦略立案

#### Champion獲得戦略
1. 技術リーダーの特定
2. ビジネス価値の訴求
3. 成功体験の提供（PoC）

#### Coach確保の方法
1. **早期アプローチ**
   - Stage1で最低1名確保
   - 技術担当者から開始
2. **信頼関係構築**
   - 定期的な情報交換
   - 相互利益の確認
3. **情報収集**
   - 社内プロセス
   - 予算サイクル
   - 競合動向

#### Blocker対策
1. 懸念事項の特定
2. リスク軽減策の提示
3. 小規模な成功事例から開始

### 4.6 Salesforce登録項目
- **Contact Roles**で各人物の役割を設定
- **Opportunity Contact Roles**にChampion/Coach/User等を記載
- **詳細はQuip**にInfluence Map文書を作成

## 5. 地図の活用と更新

### 5.1 Deal Deskでの活用
- 3つの地図を提出し、案件の実現可能性を評価
- リスク要因の特定と対策立案

### 5.2 WPSでの活用
- Influence Mapを基に攻略戦略を策定
- Champion/Coachとの連携計画

### 5.3 更新ルール
- **週次**：新情報があれば即座に更新
- **Stage移行時**：必須更新
- **月次レビュー**：SAと共同で精査

### 5.4 共有方法
- **Quip**：最新版を常に維持
- **Salesforce**：要約とリンクを記載
- **定例会議**：重要な変更を報告

- [ ] 主要ステークホルダーとの関係構築
